# AWS 시작하기
## AWS란?
### 클라우드 컴퓨팅(cloud Computing)
데스크탑 컴퓨터를 사용하려면 본체전언이 켜져있어야한다.<br>
웹 서비스를 운영하고 있다면 서비스를 실행하고 있는 컴퓨터의 본체를 관리해야할것이다.
만약에 본체가 꺼지게 된다면 사용자가 웹사이트에 접속하지 못하게 될것이다.

이상의 문제를 해결하기 위한 개념이 클라우드 컴퓨팅이다. 즉 서버나 스토리지, 데이터베이스 등 **컴퓨터 자원을 필요한 만큼만 원격으로 빌려서 사용할 수 있도록 하는 기술**을 의미한다. AWS, MS Azure, GCP 등을 예로 들 수 있는데, 가장 시장 점유율이 높은 AWS를 기준으로 수업한다.

### AW
처음에는 그냥 웹 사이트 호스팅하는 정도로 쓰였는데 오늘날에는 DB, Storage, AI등 다양한 서비스를 제공하고 있다. 즉 기업은 AWS에서 컴퓨터 자원을 빌려서 사용하고, 사용량에 따라 비용을 지불하는 형태로, 기업은 운영 비용을 줄이고, AWS 자체가 발전함에 따라 신기술을 빠르게 동비할 수 있다는 장점이 있다.

### 프로젝트 배포할건데 굳이 AWS 학습 따로하는 이유
1. 시장점유율 : 24년 기준 과학기술정보통신부 조사에 따르면 클라우드 서비스 이용하는 국내 기업의 60%정도가 AWS 활용하는 것으로 나온다.
2. 다양한 서비스와 통합 옵션 : 컴퓨팅, 스토리지, DB, AI 머신러닝 등 다양한 서비스를 제공하고, 쉽게 연동이 가능하다는 점에서 복잡한 프로젝트를 효율적으로 운영가능하다(다만 서비스 종류가 너무 많아서 AWS 자체에 대한 학습이 요구된다는 점이 좀 단점이다.)
3. 글로벌 인프라 : 전세계적으로 사용가능

### AWS 리전(region)
region은 사전적인 의미는 지역이지만, AWS에서의 개념적 정의는 이하와 같다.<br>
`AWS가 전 세계에서 데이터센트를 클러스팅하는 물리적 위치`<br>
쉽게 생각해 `AWS가 컴퓨터들을 실제로 설치해둔 위치`라고 생각해도 무방하다.
이상에서 말한 것 처럼 개발자나 기업들은 클라우드 컴퓨팅 서비스에서 사용자가 컴퓨터를 빌리게 되는데, 이렇게 AWS에서 빌려준 컴퓨터들은 실제로 만져볼 수는 없지만 전세계에 물리적으로 어딘가에 설치가 되어있다. 그런데 사용자들이 모여있는 지역을 중심으로 여러 위치에 데이터센터를 설치하게 되는데, 이를 리전이라 표현한다.

1. 특징
  1. 전세계적으로 다양한 리전을 보유하고 있다. 그리고 어느 지역의 컴퓨터를 빌릴 것인지는 사용자가 직접 선택이 가능하다.
  2. 리전마다 고유 코드가 배정되어있다. 종종 리전의 코드값을 사용할 일이 있으므로 알아두는 것이 좋다.(그리고 default로 만들면 버지니아북부로 잡히기때문에 자주 설정을 건려줘야 한다.)

2. 리전 선택의 기준 : 서비스를 주로 사용하는 사용자들과 가까운 위치
  - 사용자가 웹 사이트에 접속할 때에는 네트워크 통신을 이용한다. frontend에서 backend, 그리고 DB로 연결되는 구조를 우리는 생성했다. 그런데 한국에서 브라우저로 접속했는데 backend가 미국 버지니아 북부에 있고, db가 오사카에 있으면, 부산에서 브라우저로 접속했다가 버지니아 북부 찍고, http method 요청이 무너지 확인한 다음에 오사카에 있는 db를 가서 CRUD를 수행하고, 다시 버지니아 북부찍고 부산으로 온다.<br>
  인터넷이 연결된 선을 통해서 신호를 주고 받으면서 웹사이트와 상호작용이 이루어지는데, 물리적인 거리가 멀어질수록 신호를 주고 받는 시간이 오래 걸리게된다.(핑이라고 하기도 한다.)

# ec2로 백엔드 서버 배포하기
## 필수개념 및 EC2
위에서 ec2 검색 - 가상홈 클릭 - ec2 대쉬보드로 넘어감<br>
ec2로 배포할거임

### 배포
- 기능 구현이 끝나고, 테스트도 완료했으니까 배포하자는 표현이 나오는데, 여기서 말하는 배포란 **구현한 서비스를 서버나 클라우드 환경에 업로드**하는 것을 의미한다. 서비스를 배포하면 사용자가 인터넷을 통해 해당 서비스를 이용할 수 있다(이전까지는 localhost:8080에서만 가능).

일반적으로 웹 서비스를 개발할 때 로컬호스트라는 '자신의 컴퓨터 주소'를 사용한다. 근데 외부에서는 접속할 수가 없기 때문에 클라우드 환겨엥 웹서비스를 배포해야한다. 예를 들어 EC2에 웹 서비스를 배포하면 다른 컴퓨터에서도 접근할 수 있는 퍼블릭 IP 주소를 획득하게 된다. 사용자는 인터넷에서 이 IP 주소를 사용해서 접속이 가능해진다.

### IP 주소
인터넷에서 **특정 컴퓨터를 가리키는 주소**로, 모든 컴퓨터에 IP 주소가 부여되어있다. IP주소는 이하처럼 숫자와 점으로 이루어진다.<br>
`13.250.15.132`와 같은 방식

### 포트(port)
**한 컴퓨터 내에서 실행되는 특정 프로그램의 주소**<br>
`13.250.15.132:8080`이라고 하면 특정 컴퓨터에서 8080 프로그램을 실행했다는 의미이다.
 즉, 웬만하면 springboot 프로젝트를 실행했다고 볼 수 있다. :5173이라면 vite 프로그램을 실행했다는 뜻이 된다. 그러면 : 앞부분에 우리는 보통 localhost라는 표현을 썼는데, 이는 매번 자기 컴퓨터 IP주소를 외우기 귀찮으니까 환경변수로 자기 컴퓨터는 localhost라는 변수에 할당했개 때문이다.

 - 잘 알려지는 port : 0 ~ 65535번까지 있는데, 이 중 0~1023번까지는 특정 용도로 사용하도록 권장되어있는데, 이렇게 역할이 정해진 포트를 well-known port라고 한다.

  1. 22 : ssh 방식으로 통신시 사용, ssh는 https://를 의미한다.
  EC2에 컴퓨터를 빌려놓고 여기에 접속할때 22번, Secure Shell의 축약어
  2. 80 : HTTP 방식으로 통신할때 사용, 포트번호가 입력되지 않으면 기본적으로 80
  3. 443 : https:// 방식 통신을 할 때 사용한다. http://naver.com으로 접속하면 80번 포트로 요청을 보내고 https://naver.com로 접속하면 443번 포트로 요청한다.
  그럼 22랑 다른게 뭐냐?
- 참고 : 꼭 지켜야하는 것은 아니고 권장사항이다.

### EC2
- 정의 : 원격으로 접속해 사용할 수 있는 컴퓨터를 빌려주는 서비스

### EC2 인스턴스
- 정의 : EC2라는 서비스에서 빌리는 컴퓨터 한 대를 의미한다. 컴퓨터를 구매할 때 그래픽 통화로 할지, 용량이 큰 걸 할지 등 옵션을 선택할 수 있는데, EC2에서도 동일하다.
  1. os 이미지 : ec2 내에 탑재된 운영체제를 의미한다. windows나 mac도 있고, 가장 가볍게 쓰기 위해서는 리죽스 기반의 ubuntu를 쓴다.
  2. 인스턴스 유형 : 컴퓨터 사양을 의미한다.
  3. 스토리지 : 인스턴스 역시 컴퓨터이기 때문에 파일을 저장할 용량이 필요하다(우리 springboot 프로젝트를 올릴 용량은 있어야겠다.)

### EC2 사용이유
1. 관리 부담 절감 : 정전 등 예상치 못한 상태가 벌어졌을 때 웹 서비스에 접근하지 못하게된다. 그래서 막 서버실에 에어컨 켜놓고 있다.
2. 쉬운 보안 설정 : 개인이 전부 다 보안설정해두는게 어려울 수 있는데 아예 템플릿으로 AWS에서 제공하기 때문에 배우기에 복잡해보이지만 해두면 웬만한 수준의 보안은 할 수 있다는 장점이 있습니다.

### EC2 세팅 - 보안 그룹

![alt text](image-1.png)

보안 그룹이란 AWS 클라우드에서의 네트워크 보안을 의미한다.<br>
FC2 인스턴스를 집이라고 가정하고, 보안그룹은 집 바깥쪽에 있는 울타리 및 대문이라고 생각할 수 있겠다. 인스턴스 내부로 들어오기 전에 특정 요청이 접근해도 되는 요청인지 아닌지를 검사하는 형태라고 볼 수 있다.

이러한 규칙은 Inbound / Outbound로 나뉘어져있다.
1. inbound : 외부에서 EC2 인스턴스로 보내는 트래픽
2. outbound : EC2 인스턴스에서 외부로 나가는 트래픽을 의미한다. 여기서 어떤 트래픽만 허용할지 설정하는 것이 가능하다.

- 보안 그룹 설정
외부에서 EC2로 접근할 포트는 _22번 포트_ 와 _88번 포트_ 라고 가정하고 이 두가지에 대한 인바운드 보안 규칙을 설정할 예정이다. 22번 포트는 우리가 EC2에 원격 접속할 때 쓰는거니까, 외부에서 인터넷으로 EC2로 들어가니까 꼭 필요하다. 그리고 80번 포트는 백엔드 서버를 띄울거니까 RESTful API의 특징상 HTTP 메서드들 관련 요청이 들어갈 것이다.

### EC2에 백엔드 서버 올리기

- 터미널 띄우는 것까지 했다.(EC2 인스턴스 연결)
- 지금 그러면 우분투 os가 깔려있는 EC2에 연결을 했다고 보면 된다.
- 그럼 이 컴퓨터에 저희는 SpringBoot 프로젝트를 설치해야하고, 이 프로젝트를 실행시켜야 한다. intellij에서는 ctrl + f10 눌러서 실행시켰지만 지금 우리가 빌린 컴퓨터 내에서는 IDE가 존재하지 않기 때문에 명령어를 통해서 실행 시켜야겠다.

- 그리고 여기에는 Java도 안 깔려있다. 우리는 여태까지 프로젝트 생성시에 Java 버전을 고정시켜두는 방법을 써왔지 os 전체에 Java를 설치하는 명령어를 배우지 않았었다.
- java 설치 명령어
```cmd
sudo apt update

sudo apt install openjdk-17-jdk -y

java -version
```

이제 예시로 올려둔 프로젝트를 clone 하겠다.
```
git clone https://github.com/maybeags/ec2_springboot_sample.git
```

cloning이 끝났다면 루트 프로젝터에 들어갈 필요가 있다.
```cd
cd ec2_springboot_sample
```

현재 EC2 빌리고, 보안그룹 설정, EC2로 연결해서 Java설치, 프로젝트 설치한 후에, root 프로젝트 디렉토리로 들어가는 것까지 학습

그런데 application.properties가 없는 상태이다. 여기에 우리는 DB 비밀번호라든지 ddl 설정이라든지 다 들어가있었는데 없는 상태로 올렸다. 그 이유는 client-id와 같은 민감한 정보들이 다 들어가있기 때문에 Git으로 버전관리를 하지 않는것이 일반적이기 때문이다.
그래서 이러한 파일들은 별도로 올리거나, 그것도 아니면 EC2에서 직접 만드는 것이 편하다.

이번에는 .properties가 아니라 .yml로 만들것이다.
그런데 특정 경로에 만들어야 하기 때문에 cd 명령어를 통해서 루트 프로젝트/src/main/resources까지 들어갔다.
```cmd
cd src/main/resources
```

그 다음 application.yml을 만드는 명령어로 들어갈것이다.

```
vi application.yml
```
복사 붙여넣기 하는 법, ctrl + c, ctrl + shift + v

```yml
server:
  port: 80
```
tab이 아니라 스페이스 bar 2번임에 주의<br>
다 입력하고 난 다음엔 esc누르고 `:wq`를 쓰면 원래대로 빠져나온다.

```
cat application.yml
```
텍스트파일의 내용을 읽을 수 있다.

properties 기준으로 하면

```properties
server.port: 80
```

서버 실행용 스크립트 작성하겠다. 먼저 루트 프로젝트로 이동해야한다.

```
cd ../../../ -> ec2_springboot_sample 폴더까지 나와야한다.
```
현재 경로는 그러면

`ubuntu@ip-172-31-13-196:~/ec2_springboot_sample4`
이상의 경로에서 이제 CLI로 빌드 및 실행시키는 부분으로 넘어가야 한다.

```
./gradlew clean build # 기존 빌드된 파일을 삭제하고 새롭게 JAR 파일로 빌드함
#은 주석
```

```
chmod +x gradlew
./gradlew clean build # jar파일을 만들어준다.
```

- 중지 전 나의 public ip :http://54.180.128.16/

EC2 인스턴스를 중지하고 다시 시작할 때마다 public ip의 주소가 변경된다. 이렇게 주소가 변경되면 기존 사용자가 서버 껐다가 켤 때마다 ip 주소가 바뀌기 때문에 서비스에 접속하지 못할 수 있으므로 AWS에서는 탄력적 IP를 제공하여 변경되지 않도록 지원한햐다.

### 탄력적 IP
- AWS 공식에서는 탄력적 IP 주소란 동적 클라우드 컴퓨팅을 위해 고안된 정적 IPV4 주소를 의미한다.
- 바뀌지 않는 고정된 IP 주소
- EC2 인스턴스를 생성하면 public ip 주소를 할당받는데, 이것은 임시 ip이므로 EC2 인스턴스를 유지 보수등의 문제로 정지했다가 다시 재시작하게 되면 ip주소가 바뀌게 된다. 그러면 frontend는 기존 backend와 통신을 할 수 없게 될 것이다.(.env 파일에 VITE_API_URL='http://localhost:8080';)으로 잡아놨던 부분에서 localhost 부분을 ip 주소로 갈아끼워야하는데 이걸 고정된게 수정해야 한다는 의미

### 비용이 나가지 않게 EC2 삭제하는 법
안쓰는 탄력적 IP를 잡고 있으면 돈이 계~~~속 나간다.

1. 인스턴스 종료
  - 딸려있는 것들 삭제
  - 보안그룹은 디폴트는 삭제하지 말것
  - 키 페어삭제
  - 참조하고 있는 인스턴스가 있을 때 먼저 삭제할 수 없다. 즉 cascading 불가
2. 탄력적 IP Release
  - 이건 정확하게 얘기하면 탄력적 ip와 인스턴스간의 연결을 끊는게 아니라 탄력적 ip 자체를 날린다는 의미이다. 그런데 왜 삭제가 아니냐 하면 이 ip 주소값을 전세계의 누군가가 가져가지고 다시 재사용하게 될 수도 있기 때문이다.
3. 인스턴스 삭제
  - 위에서 삭제했는데 대시보드 보니까 반영이 늦게 되어있었다. 하지만 `실행 중`이 아니라면 돈 들지 않기때문에 신경쓰지 않는다.
